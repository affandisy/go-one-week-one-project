

# Padel Court Booking System

A comprehensive, enterprise-grade booking system designed specifically for Padel Court management. This application handles the entire lifecycle of court reservations, from dynamic pricing configurations and concurrency-safe booking logic to mock payment processing and PDF e-receipt generation.

## Project Architecture

This project adopts a decoupled architecture, separating the backend API from the frontend client. 
- **Backend:** Built with Golang utilizing Clean Architecture principles (Repository, Service, Handler layers) to ensure maintainability and testability.
- **Frontend:** Built with the latest Svelte 5 (utilizing Runes for reactivity), SvelteKit, and Tailwind CSS for a highly responsive user interface.

---

## Key Features

### Phase 1: Foundation & Master Data
- **Authentication & Authorization:** JWT-based authentication with Role-Based Access Control (RBAC) distinguishing between `customer` and `admin` roles.
- **User Management:** Profile updates and secure password changes utilizing bcrypt hashing.
- **Court Management:** CRUD operations for physical padel courts (Indoor/Outdoor, Active/Maintenance status).
- **Dynamic Pricing Rules (Surge Pricing):** Granular pricing configurations based on day types (weekday/weekend) and specific time slots using multipliers.

### Phase 2: Booking Core & Concurrency
- **Real-Time Availability Engine:** Dynamic calculation of available slots and prices based on existing reservations and pricing rules.
- **Zero Double-Booking Guarantee:** - Application level: Golang database transactions (`BeginTx`).
  - Database level: PostgreSQL Partial Unique Indexes to strictly prevent concurrent conflicting inserts.
- **Time-Locked Reservations:** Bookings are initially placed in a `locked` state for 10 minutes to allow for payment.
- **Automated Background Jobs:** A Go routine runs continuously to sweep and automatically `expire` locked bookings that exceed the 10-minute payment window.
- **Cancellation Policy:** Enforced business logic restricting customer cancellations to a minimum of 24 hours prior to the playing time.

### Phase 3: Payment, Invoicing & Analytics
- **Mock Payment Gateway:** Simulated endpoint to transition bookings from `locked` to `paid`.
- **Automated PDF E-Receipts:** Server-side generation of downloadable PDF tickets using `gofpdf` upon successful payment.
- **Admin Dashboard Analytics:** Aggregated statistics for Daily/Monthly/Yearly Revenue, Active Bookings, and Today's Occupancy Rate.

## Installation & Setup

### 1. Database Setup
Create a new PostgreSQL database for the application:
```sql
CREATE DATABASE padel_booking_db;
```

### 2. Backend Setup
Navigate to the backend directory, install dependencies, and run the server.

```bash
cd backend
go mod tidy
```

Update the database connection string in `config/database.go` with your PostgreSQL credentials.

Generate the Swagger API documentation:
```bash
swag init -g cmd/main.go
```

Start the backend server:
```bash
go run cmd/main.go
```
*The backend API will run on `http://localhost:3000`.*
*Access the Swagger UI at `http://localhost:3000/swagger/index.html`.*

### 3. Frontend Setup
Navigate to the frontend directory, install dependencies, and start the development server.

```bash
cd frontend
npm install
npm run dev
```
*The frontend application will be accessible at `http://localhost:5173`.*
